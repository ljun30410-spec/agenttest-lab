AgentTest Lab

面向「AI 问答 / Agent」类系统的轻量级测试平台，包含 AgentTest Lab（接口回归）与 Prompt Diff Lab（Prompt A/B 对比）两个工具。

Agent API 回归 Prompt A/B 对比 批量执行 + 并发 单次 / 多轮对话 AI 裁判断言 Agent Trace 测试 AI 生成 / 转换用例 历史对比 / 回归

工具简介

这是一套本地运行的网页工具，覆盖 AI 应用测试的两个关键环节：

工具地址测什么
AgentTest Lab/对已部署的 Agent API 做批量回归：curl + 断言 + Trace 工具链
Prompt Diff Lab/prompt_diff.html对同一批问题做 A/B Prompt（及模型）对比，评估迭代效果

推荐工作流：

  • Prompt 迭代阶段 → 用 Prompt Diff Lab 快速对比两套 Prompt
  • Prompt 定稿、接入 Agent → 用 AgentTest Lab 测真实接口与工具链
  • 发版 / 改代码后 → AgentTest Lab 全量回归，与历史报告对比

技术构成非常轻量：

  • 后端：server.js，CORS 代理、报告存储、AI 模型调用、Prompt Diff API。
  • 前端：test_tool.html、prompt_diff.html + test_tool.helpers.js，纯原生 JS，无需构建。
  • AI 能力：通过 Vercel AI SDK 调用 OpenAI 兼容模型，默认面向腾讯云 TokenHub。
  • 状态保留：两页顶部可互相切换；表单输入与最近报告在页面切换后自动恢复。

它解决什么问题

传统接口测试用「精确匹配」来断言，但 AI 的回答是不确定的自然语言，每次措辞都不一样，没法逐字比对。本工具针对这一特点提供了多层断言能力：

能力说明类型
基础校验状态码、响应时间、内容长度规则
关键词软断言命中率达到阈值即通过，不要求逐字全中规则
Agent Trace 断言验证真实工具调用链、步骤顺序和工具耗时规则
AI 裁判断言用大模型判断回答是否语义正确、是否安全合规模型

快速开始

  1. 确认环境

    本机安装 Node.js（建议 v18 以上）。首次使用请在项目目录安装依赖：

    npm install

  2. 启动服务

    在项目目录下运行：

    cd "AI agent 测试"
node server.js

    看到 Server running on http://localhost:5001 即启动成功。

    💡 建议在你自己的终端里启动，这样服务会一直常驻，不受其他程序影响。

  3. 打开页面

    工具地址
    AgentTest Lab（接口回归）http://localhost:5001/
    Prompt Diff Lab（Prompt 对比）http://localhost:5001/prompt_diff.html
    本使用手册http://localhost:5001/使用手册.html

    两个工具页面顶部有导航栏，可随时切换；切换后已填写的内容和最近报告会自动恢复。

① 配置接口

  1. 粘贴 cURL 命令

    在浏览器开发者工具或抓包工具里，对被测接口「复制为 cURL」，粘贴到 Curl 命令 框。工具会自动解析出 URL、请求头和请求体。

    ⚠ cURL 里通常包含你的登录态（Cookie / Token），请用本人账号的正常请求，测试过程中不要再用该账号手动发起 AI 请求，以免互相干扰。

  2. 设置问题字段路径

    问题字段路径：测试问题写入请求体的哪个位置。填 auto 会自动识别 message.content / question / query 等常见字段；也可手动指定，如 message.content。

  3. 设置 knowledgeId 字段路径（可选）

    若接口需要知识库 ID，填写其字段路径（支持嵌套，如 message.extraData.knowledgeId）；不需要则留空或填 none。用例里的 knowledgeId: null 表示沿用 cURL 请求体中的原值，不会覆盖。

  4. 设置 conversationId 字段路径（多轮模式）

    打开 多轮对话模式 后会显示 conversationId 字段路径。默认是 conversationId，用于把同一个真实会话 ID 写入后续轮次。

    ⚠ 工具不会伪造新的 conversationId。它只会复用 cURL 里已有的真实 ID，或从首轮流式响应里的 conversationId / conversation_id / conversation.id 提取真实 ID 后用于后续轮次。如果接口要求先创建会话，请先在产品里复制一条有效会话的 cURL，或确认首轮接口会返回真实会话 ID。

② 准备测试用例

共有 四种方式，可任选其一，结果都会进入下方「测试用例」文本框，运行前都可以再编辑。

方式 A：手动填写 / 粘贴

支持 JSON 数组 或 Markdown 表格 两种格式。默认是单次对话模式，JSON 示例：

[
  {
    "id": "FT-001",
    "title": "基础问答",
    "type": "功能测试",
    "priority": "P0",
    "question": "重低音测试的作用是什么？",
    "knowledgeId": null,
    "expected": {
      "status_code": 200,
      "keywords": ["重低音", "测试", "音频设备"],
      "min_length": 10,
      "max_time": 50
    },
    "validation_points": "答案与资料一致"
  }
]
knowledgeId: null 表示不从用例覆盖知识库 ID，优先使用你粘贴的 cURL 请求体里的真实值。不要把示例占位符写成真实 ID。
💡 单次模式的 JSON 模板会包含 FT-001、BT-001 和 TR-001。其中 TR-001 是 Agent Trace 示例，用来演示如何校验 web_search 等工具调用。

方式 B：上传文件 无需模型

点击 📁 上传用例文件，支持 .md / .json / .csv / .xlsx。CSV/Excel 的列名支持中英文别名（如「输入问题 / question」「关键词 / keywords」「优先级 / priority」），顺序随意。

💡 不知道该传什么格式？点旁边的 下载模板（.md / .json / .csv），下载示例文件、填好再传回来即可。模板会根据当前「单次 / 多轮」模式自动切换。
📎 .xlsx 解析需要联网加载一次解析库；纯离线环境请用 .csv / .json / .md。

方式 C：AI 转换用例 模型

展开 🪄 AI 转换用例（自然语言 → 标准格式），把你已经写好的自然语言用例粘进去，点 🪄 转换为标准用例。这个功能只做结构化整理，不会发散新增用例：输入几条自然语言用例，就必须输出几条标准 JSON 用例。

💡 推荐一行写一条用例；如果某条用例很长，可以用单独一行 --- 分隔。系统不会按普通句号拆分，避免把同一条用例误拆成多条。
💡 如果自然语言里写了“触发联网搜索 / 调用 web_search”，会补充 expected.trace.required_tools；写了“不应联网 / 不要调用 web_search”，会补充 expected.trace.forbidden_tools。
⚠ 如果模型返回数量和输入条数不一致，转换会直接失败并提示重试，而不是把多出来或少掉的用例填入文本框。

方式 D：AI 生成用例 模型

展开 🤖 AI 生成用例（草稿），用自然语言描述需求（或粘贴整段需求文档），设置生成条数，点 ✨ 生成用例。模型会自动起草覆盖功能 / 边界 / 异常 / 安全等维度的用例，填入文本框。

⚠ AI 生成的是草稿，结果存在不确定性。请务必人工审核 / 微调（尤其是关键词和 knowledgeId）后再运行。此功能复用下方「AI 断言」的模型配置。

两个参数

参数说明
并发数单次模式下同时执行的用例数，建议 3-5；设为 1 即串行执行。多轮模式固定为 1，不支持并发
关键词命中率阈值 (%)软断言阈值：命中率达到该值即通过，未命中的关键词只记警告，不直接判失败（默认 60%）

单次 / 多轮对话模式

工具现在把单次对话和多轮对话显式拆开，避免把 question 和 turns[] 混在一起执行。

单次对话模式（默认）

  • 适合批量验证独立问题，例如知识库问答、边界输入、安全问题等。
  • 用例使用顶层 question 字段。
  • 支持并发执行，按「并发数」配置运行。
  • 如果用例里误放了 turns[]，工具会提示；继续运行时只使用 question。

多轮对话模式

  • 打开 多轮对话模式 开关后启用。
  • 用例必须使用 turns[]，同一个场景内按第 1 轮、第 2 轮、第 3 轮顺序执行。
  • 固定串行执行，并发数会被禁用，避免上下文顺序错乱。
  • 失败重跑会重跑整个失败场景，而不是单独重跑某一轮。
  • 报告会展示每轮问题、状态、耗时、失败原因和原始响应。

多轮 JSON 示例：

[
  {
    "id": "MT-001",
    "title": "同一上下文多轮追问",
    "type": "多轮对话",
    "priority": "P0",
    "knowledgeId": null,
    "conversation": { "mode": "same_context" },
    "turns": [
      {
        "question": "请介绍这份资料的核心内容",
        "expected": {
          "status_code": 200,
          "keywords": ["核心", "内容"],
          "min_length": 10,
          "max_time": 50
        }
      },
      {
        "question": "刚才提到的第二点展开说说",
        "expected": {
          "status_code": 200,
          "keywords": ["第二点"],
          "min_length": 10,
          "max_time": 50
        }
      },
      {
        "question": "总结一下我们刚才聊了什么",
        "expected": {
          "status_code": 200,
          "keywords": ["总结"],
          "min_length": 10,
          "max_time": 50
        }
      }
    ],
    "validation_points": ["多轮问题应保持同一上下文"]
  }
]
⚠ 多轮会话是否真的落在同一个产品会话里，取决于被测接口的 conversationId 机制。若响应出现 {"code":7,"msg":"未发现数据"}，通常表示 cURL 中的会话 ID 已失效、被测产品要求先创建会话，或 conversationId 字段路径 配错。

③ 配置 AI 能力（可选）

「AI 断言（裁判模型）」、「AI 生成 / 转换用例」和 Prompt Diff Lab 共用同一套模型配置，底层通过 Vercel AI SDK 调用腾讯云 TokenHub（OpenAI 兼容）。推荐用服务端 .env 配置，前端三个输入框可全部留空。

方式一（推荐）：服务端 .env

复制 .env.example 为 .env，填入你的 Key：

TOKENHUB_API_KEY=sk-your-key-here
TOKENHUB_BASE_URL=https://tokenhub.tencentmaas.com/v1
TOKENHUB_MODEL=deepseek-v4-flash

方式二：前端临时填写

字段说明
Base URL留空即使用服务端 .env / judge.config.json；默认 https://tokenhub.tencentmaas.com/v1
模型名留空即使用服务端 .env / judge.config.json；默认 deepseek-v4-flash
API Key你的密钥（仅存于浏览器 sessionStorage，关闭标签页即清除）

填完点 🔌 测试连接 验证是否可用。打开顶部开关后，每条用例（多轮场景中每一轮）都会经过 AI 裁判判定。

🔐 配置优先级：前端完整填写 API Key 时使用前端配置；若前端没有填写 API Key，则忽略前端 Base URL / 模型名，完全使用服务端 .env（TOKENHUB_*）或 judge.config.json，避免 Key 与 Base URL 错配。
💡 性能提示：开启 AI 断言会因为多一次模型调用而变慢。工具内置了结果缓存——相同问题与回答不会重复请求模型，可缓解耗时。

④ 执行测试

点击 🚀 开始测试。顶部会出现一条非阻塞进度条，显示进度百分比、当前用例 ID，并提供 ⏹ 取消 按钮，随时可中断。

单次模式会按并发数执行；多轮模式固定串行，同一个场景内按 turns[] 顺序请求。执行完成后自动生成报告，并把本次结果保存到历史。

⑤ 看懂报告

报告区顶部是汇总统计（总数 / 通过 / 失败 / 警告等）。每条用例可展开看详情：请求、响应、各断言项的命中情况、AI 判定理由等。

详情中有四个 Tab：请求的 CURL、原始数据、纯文本、Trace。流式 SSE 响应会被归一化为纯文本用于关键词断言和 AI 裁判；响应内嵌的 trace/debug 数据会被解析成步骤表格用于排查链路。

搜索 / 筛选按关键字、用例类型、结果（通过 / 失败 / 警告）过滤

🔁 重跑失败只重新执行失败的用例，省时间

⑥ 历史与对比

点击 📚 历史与对比 查看保存过的历次运行。选择两次运行进行 🔍 对比，工具会逐用例列出差异并分类：

标记含义
回归（变坏）上次通过、这次失败 —— 重点关注
修复（变好）上次失败、这次通过
变化状态、关键词命中、内容、性能、AI 判定或 Trace 链路有变化
新增 / 移除用例集发生了增删
对比不同的数据集时会给出提示，因为用例不一致时差异结果仅供参考。
💡 页面顶部的导航可在 AgentTest Lab 与 Prompt Diff Lab 之间切换；curl、用例、Prompt 配置和最近报告都会保留，无需重新填写。

Prompt Diff Lab 简介

Prompt Diff Lab 用于在 不经过 Agent 接口 的情况下，直接对 LLM 跑 A/B 对比，专门评估 System Prompt 或模型变更后的回答质量。

访问地址：http://localhost:5001/prompt_diff.html，或在 AgentTest Lab 页面顶部点击 Prompt Diff Lab 切换。

对比维度说明
并排回答同一问题下 A/B 两版回答左右对照
通过率启用 AI 裁判后，A/B 各自 pass 率
均分AI 裁判 0–100 分均值及 B 胜场数
关键词命中JSON 用例带 keywords 时统计命中率
耗时 / 字数每条回答的响应耗时与字符数
Prompt Diff Lab 与 AgentTest Lab 的 AI 断言共用同一套模型配置（.env / judge.config.json / 前端临时 Key）。

Prompt Diff Lab 使用步骤

  1. 配置模型

    推荐在服务端 .env 配置 TOKENHUB_API_KEY，前端 Key 可留空。也可在前端临时填写 Base URL / API Key / 默认模型。

  2. 填写测试问题

    支持两种方式：

    • 纯文本：一行一条，或用 --- 分隔较长问题
    • JSON 数组：可带 keywords 用于关键词命中率统计
    [
  { "question": "什么是机器学习？", "keywords": ["学习", "数据"] },
  { "question": "把「今天天气不错」翻译成英文。" }
]

  3. 配置 A / B 版本

    版本 A（基线）：当前线上或上一版 System Prompt，可选指定模型。
    版本 B（候选）：待评估的新 Prompt，也可换模型做对比。

    模型留空则使用「默认模型」配置。

  4. 开始对比

    勾选「启用 AI 裁判」可对每条回答做语义评分（更慢但更全面）。点击 开始对比，工具会对每个问题分别跑 A 和 B 各一次。

    💡 可点 加载示例 快速体验：3 条问题 + 简短 vs 详细两套 Prompt。

Prompt Diff Lab 看懂对比报告

对比完成后，页面展示汇总指标与逐题表格：

汇总指标含义
A / B 通过率启用 AI 裁判时，各自 pass 的题数占比
A / B 均分AI 裁判分数均值
A / B 关键词命中期望关键词在回答中的平均命中率
B 胜场按裁判分数 B 高于 A 的题数

逐题表格中，「对比」列会标出 A 更优、B 更优、平局，或按字数差异提示「A/B 更长」。

报告自动保存为 reports/prompt-diff-{timestamp}.json，受与 AgentTest Lab 相同的报告保留策略管理（默认最近 50 份）。

⚠ Prompt Diff 测的是「模型直接回答」，不包含 Agent 的工具调用、知识库检索等环节。Prompt 定稿后，仍需用 AgentTest Lab 对真实接口做回归。

断言机制说明

一条用例最终是否「通过」，由以下判定共同决定：

  • 状态码 / 长度 / 耗时：硬性校验，不满足直接失败。
  • 关键词：软断言。命中率 ≥ 阈值即通过；未命中的词会记为警告；命中率不足会判失败。
  • AI 裁判（若开启）：模型给出 pass / score / reason。判不合格则该用例失败，并附上理由。
  • Trace 断言（若配置）：校验工具调用、步骤顺序和耗时阈值。

断言内容优先使用归一化后的 normalized_content。对于 text/event-stream 流式响应，工具会从 data: 行中提取 content / text / answer / message 等文本字段。

设计取向：尽量减少「措辞不同就误判失败」的噪音，让失败更聚焦在真正的问题上。

Agent Trace 测试

Trace 测试用于验证 Agent 是否走了预期链路，例如是否触发联网搜索、是否按「检索 → 工具调用 → 回答」顺序执行、某个工具是否耗时过长等。它主要解决“答案看起来对，但 Agent 实际有没有调用正确工具”的问题。

Trace 数据来源

当前版本只解析被测接口响应内嵌的 trace/debug 数据，不额外调用独立 trace 查询接口。工具会先识别严格的 tool_call start/end 事件；如果没有，再从 JSON 或 SSE 的 data: payload 中递归查找这些常见字段作为兼容 fallback：

  • trace / debug.trace
  • steps / events / spans
  • tool_calls / toolCalls

真实工具调用链规则

当响应里存在事件流格式的 tool_call start/end 事件时，工具会优先按真实工具调用链规则解析：

  • 过滤所有 reasoning_start / reasoning / reasoning_end 模型思考文本，它们只代表预规划，不计入真实调用。
  • 过滤 tool_call_intent，它只代表模型准备调用工具的意图。
  • 只保留同时具备唯一 toolCallId、tool_call start + end、success=true、latencyMs 和 output 的成功工具调用。
  • 按事件时间或出现顺序升序排列，输出标准 filter_explain + real_tool_chain JSON。

Trace Tab 中会展示如下标准 JSON，便于直接复制给程序处理：

{
  "filter_explain": "说明过滤了哪些内容，思考链路和实际调用的差异",
  "real_tool_chain": [
    {
      "sequence": 1,
      "tool_call_id": "唯一ID",
      "tool_name": "工具名称",
      "request_params": "工具入参",
      "latency_ms": 1234,
      "execute_status": "success",
      "result_summary": "返回结果摘要"
    }
  ]
}
如果响应里没有内嵌 trace，报告的 Trace Tab 会显示「未发现 trace 数据」。只有用例配置了 expected.trace 时，trace 缺失才会导致失败。

Trace 断言格式

在用例的 expected.trace 下配置断言：

{
  "id": "TR-001",
  "title": "资料不足时应触发联网搜索",
  "type": "Trace测试",
  "priority": "P1",
  "question": "资料不足时请结合联网搜索回答今天的热点新闻",
  "knowledgeId": null,
  "expected": {
    "status_code": 200,
    "keywords": ["新闻"],
    "trace": {
      "required_tools": ["web_search"],
      "forbidden_tools": [],
      "required_steps": ["retrieve", "web_search", "answer"],
      "max_total_time_ms": 15000,
      "max_tool_time_ms": 8000
    }
  }
}
字段说明
required_tools必须出现的工具名，例如 web_search
forbidden_tools不允许出现的工具名，例如某些场景禁止联网
required_steps要求按顺序出现的步骤名，不要求相邻
max_total_time_msTrace 步骤总耗时上限
max_tool_time_ms单个工具调用耗时上限

报告与回归

报告详情的 Trace Tab 会显示步骤数、工具集合、总耗时、失败步骤数，以及每一步的名称、类型、状态、耗时、输入/输出摘要。若识别到真实工具调用链，还会额外展示可直接复制解析的标准 JSON。

历史报告只保存 trace 摘要，不保存完整 raw trace，避免报告文件过大。历史对比会提示工具增删、步骤序列变化、失败步骤变化、耗时显著变化和 trace hash 变化。

报告管理

每次运行的结果以 JSON 文件保存在服务端 reports/ 目录。为避免目录无限增长，工具内置保留策略：

  • 默认只保留最近 50 份报告，多余的在保存新报告时自动清理。
  • 可通过环境变量调整上限：MAX_REPORTS=100 node server.js

安全说明

  • API Key：前端填写的密钥仅存于 sessionStorage，关闭标签页即清除，不写入磁盘、不入库。
  • 表单数据：curl、用例、Prompt 配置等存于浏览器 localStorage / sessionStorage，仅在本机浏览器中，不会提交到 Git 仓库。
  • XSS 防护：报告中所有来自接口 / 模型的内容都做了 HTML 转义后再渲染。
  • 私密配置：.env、judge.config.json 与 reports/ 已被 .gitignore 忽略，不会误提交。
  • 登录态：cURL 含账号凭证，请勿把含真实 Cookie 的内容分享给他人。

常见问题

启动时报 EADDRINUSE（端口被占用）？

说明 5001 端口已有进程在用。先结束旧进程再启动：

lsof -ti tcp:5001 | xargs kill -9
node server.js

接口报「无法解析域名 / 连接失败」？

若被测接口是公司内网地址，请确认已连上公司网络或 VPN，并保证运行服务的环境能访问该内网。工具会把常见网络错误翻译成中文提示。

开了 AI 断言但没生效？

如果使用服务端 .env，前端 Base URL / 模型名 / API Key 可以全部留空，然后点「测试连接」。如果前端手动填写，则必须保证 Base URL、模型名、API Key 属于同一服务商，避免 TokenHub Key 配到其他 Base URL 上。

AI 生成 / 转换用例失败？

生成失败多为模型输出被截断或返回非 JSON，可减少「生成条数」后重试；转换失败如果提示数量不一致，请把自然语言用例改成一行一条，或用 --- 分隔长用例。两者都需要确认模型配置可用。

遇到 401 / 403？

多为登录态过期。重新获取最新的 cURL（含新 Cookie / Token）再粘贴。报告会给出登录失效提示。

AI 断言测试连接提示 API Key 无效？

优先检查 .env 中的 TOKENHUB_API_KEY 是否正确、是否已过期。若你曾在前端填写过其他 Base URL / 模型名，建议清空前端三个输入框并硬刷新页面，让服务端完整接管模型配置。

多轮对话返回 {"code":7,"msg":"未发现数据"}？

通常是 conversationId 无效或字段路径不对。工具不会伪造会话 ID；它只会复用 cURL 里已有的真实 ID，或使用首轮响应返回的真实 ID。请检查「请求的 CURL」里的请求体：如果带了旧的 conversationId，请重新从产品里复制当前可用会话的 cURL；如果不该带会话 ID，可把 conversationId 字段路径 填成 none。

多轮问题没有落在同一个会话？

确认已打开 多轮对话模式，用例使用 turns[]，且并发输入已变成「多轮模式固定串行」。然后展开报告看每轮「请求的 CURL」：后续轮次应带同一个真实 conversationId。如果首轮响应没有返回会话 ID，工具无法凭空创建产品侧会话，需要被测接口或原始 cURL 提供有效 ID。

Trace Tab 显示「未发现 trace 数据」？

说明被测接口响应中没有内嵌可识别的真实 tool_call start/end 事件，也没有兼容的 trace / debug.trace / steps / tool_calls 等字段。若产品需要通过单独接口查询 trace，当前版本还不会自动调用；可以先让被测接口在响应或 SSE 事件里带上 trace 数据。

Trace 断言失败，但原始响应里有 reasoning？

reasoning 属于模型思考或预规划，不代表工具真的执行。工具只把成对且成功的 tool_call start/end 事件计入真实工具调用链。若只有 tool_call_intent 或 reasoning 文本，没有成功的 end 事件、latencyMs、output，会被过滤。

改了代码但页面还是旧表现？

重启 node server.js 后，用 Cmd + Shift + R 硬刷新页面，确保浏览器重新加载 test_tool.html、prompt_diff.html 和 test_tool.helpers.js。

切换到 Prompt Diff Lab 再切回来，数据丢了？

两个工具页面均支持自动保存与恢复。若仍丢失，请确认未使用无痕模式，并检查浏览器是否禁用了 localStorage。硬刷新不会清除已保存的表单，但关闭标签页会清除 sessionStorage 中的 API Key 和最近报告。

Prompt Diff 对比很慢？

每个问题会依次跑 A 和 B 各一次；若开启 AI 裁判，每条还要额外裁判两次。建议先用少量问题验证，再扩大题集。可暂时关闭 AI 裁判，仅看并排回答与字数差异。
